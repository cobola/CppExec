Docker化C++程序接口服务需求文档

一、文档概述

1.1 文档目的

本文档明确Docker化C++程序接口服务的开发需求、技术规范、部署要求及调用标准，作为开发、测试、部署全流程的依据，确保最终产品满足“极简可运行、易替换、跨平台”的核心目标。

1.2 适用范围

适用于开发人员、测试人员、运维人员，覆盖C++业务程序封装、FastAPI接口层开发、Docker打包配置、部署及接口调用全环节。

1.3 核心定位

针对轻量C++程序（计算/逻辑判断类），提供“接口层+容器化”一体化解决方案，支持本地/服务器/Docker多环境部署，降低新手开发门槛，实现单文件替换即可快速迭代业务逻辑。

二、项目背景与目标

2.1 项目背景

C++程序在计算性能、逻辑处理上具备优势，但直接提供HTTP接口存在开发成本高、跨平台兼容性差、部署配置繁琐等问题。需通过轻量接口层封装+容器化技术，解决C++程序的接口化调用与便捷部署需求，兼顾易用性与兼容性。

2.2 核心目标

- 极简可运行：仅需3个核心文件，支持一键Docker构建与启动，新手可快速上手。

- 接口化调用：提供标准HTTP接口，支持GET/POST两种方式，无需直接操作C++程序。

- 高兼容性：支持Windows/macOS/Linux多系统部署，依赖Docker实现环境一致性。

- 易扩展性：支持C++业务逻辑快速替换，接口参数、依赖库可灵活调整。

- 轻量高效：镜像体积控制在百MB级，低配服务器（1核2GB）可稳定运行。

三、功能需求

3.1 C++业务程序模块

- 交互方式：支持命令行传参接收输入，通过控制台标准输出返回结果，无多余打印信息（便于接口层捕获）。

- 参数校验：具备基础参数合法性校验能力，参数错误时输出明确错误信息并返回非0状态码。

- 可替换性：核心业务逻辑独立，支持用户替换为自定义计算/逻辑判断代码，无需修改交互机制。

- 示例功能：默认实现双浮点数加法逻辑，作为基础演示模板。

3.2 FastAPI接口层模块

- 接口能力：提供与Judge0兼容的核心接口（/execute），支持POST请求方式，接口标准对齐Judge0基础执行规范，便于集成现有Judge0适配体系。

- 参数处理：遵循Judge0接口参数格式，接收与C++程序匹配的输入参数（原a、b参数适配为Judge0规范字段，保留核心传参逻辑），转发至C++可执行文件，捕获输出结果、错误信息及退出码，同步适配Judge0的返回字段映射。

- 返回格式：统一JSON返回，兼顾原功能逻辑与Judge0兼容性，字段包含status（状态信息）、stdout（程序输出）、stderr（错误信息）、exit_code（退出码），同时保留原状态码便于问题排查，格式规范如下：
        
- 成功：{"code":200,"msg":"调用成功","status":{"id":1,"description":"Completed"},"stdout":"结果值","stderr":"","exit_code":0}
        
- C++程序错误：{"code":500,"msg":"C++程序执行失败","status":{"id":2,"description":"Runtime Error"},"stdout":"","stderr":"错误详情","exit_code":1}
        
- 接口层错误：{"code":400,"msg":"接口调用失败","status":{"id":3,"description":"Bad Request"},"stdout":"","stderr":"错误详情","exit_code":-1}
      

- 本地测试：支持直接运行Python文件启动服务，无需依赖Docker，测试端口默认使用4002。

- 安全限制：接口调用需满足双重安全校验，一是请求头携带API Key鉴权（X-API-Key），无有效Key则拒绝请求；二是限制单IP每分钟请求频率≤60次，超出频率返回限流提示，防范恶意调用。

3.3 Docker打包模块

- 基础镜像：采用Alpine 3.19，保证轻量性与兼容性。

- 自动构建：集成C++编译环境（g++）与Python依赖（fastapi、uvicorn），构建时自动编译C++代码生成可执行文件。

- 端口暴露：默认暴露4002小众端口（避免与常用端口冲突，降低端口占用风险），与接口服务端口一致，支持自定义映射但建议保留4002作为容器内端口。

- 启动命令：容器启动后自动启动uvicorn服务，监听所有IP地址，端口固定为4002，命令调整为：["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "4002"]。

3.4 部署与调用模块

- 部署命令：提供3条核心Docker命令（构建镜像、启动容器、验证状态），支持跨平台执行。

- 调用方式：支持3种接口调用方式，满足不同场景需求：

- Postman：POST请求快速测试。

- curl命令：POST请求，适用于服务器/终端场景。

- Python代码：POST请求，便于业务系统集成。
      

- 状态验证：支持通过Docker命令查看容器运行状态，确保服务正常启动。

四、非功能需求

4.1 兼容性

支持Windows 10+/macOS 12+/Linux（CentOS 7+/Ubuntu 20.04+）系统，Docker版本需≥20.10.0。

4.2 性能

单容器在1核2GB服务器上，支持每秒≥50次接口调用，接口响应时间≤100ms（不含C++业务逻辑耗时）。

4.3 易用性

核心文件无需修改即可运行，替换C++业务逻辑仅需调整3处关键点，提供明确修改指引；部署与调用步骤简洁，无复杂配置。

4.4 可扩展性

- 支持C++程序参数数量、类型调整，接口层可同步适配。

- 支持C++第三方依赖库集成，可通过Dockerfile编译命令扩展。

- 支持接口鉴权、多进程启动、目录挂载等进阶功能扩展。

4.5 稳定性

容器后台运行时，无异常退出情况；接口调用错误时，返回明确错误信息，便于问题排查。

五、技术规范

5.1 文件结构规范

核心文件统一放在同目录（cpp-docker-api/），结构如下，不可随意变更目录层级：

cpp-docker-api/
├── main.cpp       # C++业务程序（自定义业务逻辑核心文件）
├── main.py        # FastAPI接口层文件（固定模板，可微调参数）
└── Dockerfile     # Docker打包配置文件（固定模板，可扩展依赖）

5.2 技术栈要求

- C++：支持C++11及以上标准，编译后生成可执行文件（命名为cpp_app）。

- Python：3.8+，依赖库版本：fastapi≥0.100.0，uvicorn≥0.23.2。

- Docker：基础镜像Alpine 3.19，依赖工具：g++、python3、py3-pip、gcc、make。

5.3 编码规范

- C++：代码简洁可维护，参数校验逻辑前置，输出结果仅包含业务数据，无冗余日志。

- Python：接口注释完整，异常捕获全面，变量命名规范（英文小写+下划线），保持代码可阅读性。

- Dockerfile：命令按“安装依赖→设置目录→复制文件→编译→安装依赖→启动”顺序编写，减少镜像层数。

六、核心修改点说明（业务替换指南）

替换为自定义C++业务逻辑时，仅需调整以下3处，无需修改其他代码：

1. main.cpp：替换核心业务逻辑，保留“命令行传参+控制台输出结果”交互方式，同步调整参数校验逻辑（argc数量），确保输出格式可适配Judge0的stdout/stderr字段。

2. main.py：一是将接口路径改为/execute，参数与返回格式适配Judge0标准；二是添加API Key鉴权与IP频率限制逻辑；三是同步修改端口为4002，subprocess传参列表与C++程序argc/argv对应。

3. Dockerfile（可选）：若C++程序依赖第三方库，在g++编译命令后添加链接参数（如-lxxx，xxx为库名）；无需调整端口配置（已固化为4002）。

七、进阶优化需求（可选）

根据实际业务场景，可选择性实现以下优化功能：

1. 并发优化：Docker启动命令中开启uvicorn多进程（--workers 4），适配多核CPU，同时维持4002端口不变。

2. 文件挂载：启动容器时添加目录挂载（-v /本地目录:/app/data），支持C++程序读写外部文件。

3. 性能优化：将C++程序编译为动态库（.so/.dll），通过Python ctypes直接调用，消除子进程启动开销，同时不影响Judge0接口兼容性。

4. 安全优化：强化接口安全限制，除基础API Key鉴权与频率限制外，可扩展IP白名单机制，仅允许指定IP访问接口；同时禁止容器以root用户运行，降低安全风险。

5. 运维优化：集成docker-compose，实现容器自启动、日志持久化管理，配置文件中同步指定4002端口映射。

八、部署与调用说明

8.1 部署步骤

1. 构建镜像：docker build -t cpp-api:v1 .（镜像名与版本可自定义）。

2. 启动容器：docker run -d -p 4002:4002 --name cpp-api-server cpp-api:v1（端口映射保持内外一致为4002，避免冲突，容器名可自定义）。

3. 验证状态：docker ps，确认容器状态为“Up”即部署成功，可通过docker logs查看4002端口监听日志。

8.2 接口调用规范

- 接口地址：http://[IP]:4002/execute（本地IP为127.0.0.1，默认端口4002，服务器为服务器公网IP，需携带API Key请求）。

- 请求方式：POST，请求头必须携带X-API-Key字段，值为预设密钥。

- 参数格式：遵循Judge0接口标准键值对格式，核心参数与C++程序接收类型一致，同时兼容原业务参数逻辑。

九、服务器部署注意事项

1. 端口放行：服务器防火墙/安全组仅放行4002端口，建议限制访问来源IP，进一步降低端口暴露风险。

2. 依赖检查：服务器需提前安装Docker，版本≥20.10.0，避免构建失败。

3. 资源适配：低配服务器建议关闭不必要的进程，确保容器运行资源充足，同时监控4002端口占用情况，防范端口被恶意占用。

4. 长期运行：搭配docker-compose实现容器自启动，配置文件中固化4002端口映射与安全启动参数；定期更新API Key，强化鉴权安全性。

十、附则

1. 本文档需求以“极简可运行”为核心，若业务需扩展复杂功能，可补充修订需求。

2. 开发过程中需严格遵循技术规范，确保产品兼容性与可扩展性。

3. 接口调用异常时，优先排查容器运行状态、C++程序参数格式及业务逻辑。
